==================================================
1. Executive Summary
==================================================

The current branch is trying to solve a valid problem:
- determine whether a media item is publicly visible on a photobank from the customer side
- if visible, infer that the item is effectively approved/live

That product direction is reasonable.

The main weakness is not the idea, but the implementation strategy:
- the code currently relies on a large amount of bank-specific HTML scraping
- matching is still mostly title-based
- the main runner mixes crawling, anti-bot handling, matching, dry-run logging, and CSV writes in one flow

My conclusion:
- the branch is directionally correct
- the architecture is still too heuristic and too operationally complex
- for several banks, official customer-side APIs or official catalog/search channels should be preferred over scraping public portfolio pages
- for the remaining banks, the fallback should be public site search or item-page inspection, not necessarily contributor-portfolio scraping

==================================================
2. Static Code Review
==================================================

2.1. What the branch is solving

This branch adds a new non-GUI mode to markphotomediaapprovalstatus:
- public portfolio crawling
- asset extraction per bank
- matching against PhotoMedia.csv
- optional dry-run mode
- session/cookie handling for anti-bot protected banks

Primary touched area:
- markphotomediaapprovalstatus/markphotomediaapprovalstatus.py
- markphotomediaapprovalstatus/markphotomediaapprovalstatuslib/public_portfolio/*

2.2. Main strengths

- The feature target is well chosen: public customer visibility is often more relevant than internal contributor-side status.
- Per-bank adapters are the right architectural direction because each bank exposes different HTML and metadata.
- The code separates some concerns correctly:
  - browser handling
  - matching
  - config
  - session/cookies
  - bank-specific extraction
- Dry-run support is useful and operationally important.

2.3. Main weaknesses and what should be improved

1. The main runner is doing too much.

File:
- markphotomediaapprovalstatus/markphotomediaapprovalstatuslib/public_portfolio/runner.py

The main flow currently combines:
- config loading
- bank iteration
- portfolio crawling
- anti-bot retry/session handling
- dry-run logging
- record matching
- CSV persistence

This makes the feature hard to debug and hard to stabilize. The function should be split into at least:
- process_bank()
- crawl_bank_assets()
- match_bank_records()
- persist_bank_updates()
- handle_blocked_bank()

2. Automatic approval is still based on relatively weak evidence.

File:
- markphotomediaapprovalstatus/markphotomediaapprovalstatuslib/public_portfolio/matching.py

The current matching logic is mostly:
- normalized title match
- pipe-segment match
- prefix match

This is usable as a heuristic, but not strong enough to safely auto-write approvals across many banks.

Risks:
- false positives for similar title series
- truncated title collisions
- language/punctuation normalization edge cases
- overly optimistic prefix matches

Recommendation:
- only auto-approve on "strong match"
- introduce match confidence tiers:
  - strong
  - weak
  - ambiguous
  - none
- weak matches should log only, not update CSV automatically

3. PublicAsset is missing a stronger identity contract.

File:
- markphotomediaapprovalstatus/markphotomediaapprovalstatuslib/public_portfolio/models.py

PublicAsset currently has:
- bank
- url
- contributor_id
- title
- description

For many banks, URL or page metadata contains a stable asset identifier. That should be modeled explicitly if available, for example:
- asset_id
- source_type
- media_type
- maybe width/height if present

Without that, the system is forced to overuse title-based inference.

4. Portfolio scraping is likely not the best primary source for all banks.

Files:
- public_portfolio/banks/*
- public_portfolio/runner.py

Portfolio pages are often:
- lazy-loaded
- anti-bot protected
- incomplete
- operationally unstable

For many banks, public customer search/catalog endpoints are a cleaner primary source than contributor/public portfolio pages.

Recommendation:
- redesign bank strategy as:
  - API search first
  - official site search fallback
  - scraping fallback only when necessary

5. Session saver is too tightly coupled to the main automated flow.

Files:
- public_portfolio/runner.py
- public_portfolio/session.py

The main run currently attempts interactive recovery for blocked banks.
That increases operational complexity and makes automation less predictable.

Recommendation:
- make automatic mode report blocked banks
- run cookie/session capture explicitly as a separate maintenance step
- keep the automatic pipeline deterministic

6. Bank adapters are useful, but normalization logic is too fragmented.

Files:
- public_portfolio/banks/base.py
- public_portfolio/banks/*.py

Each adapter currently invents its own extraction style:
- URL slug parsing
- alt text parsing
- aria-label parsing
- anchor text parsing

That is unavoidable to some extent, but quality rules should be centralized more aggressively in BaseBankAdapter:
- minimal title quality
- title cleanup
- duplicate handling
- asset quality thresholds

7. Logging volume is decent, but diagnostic value is still lower than needed.

The code logs a lot of steps, but still does not make it easy to answer:
- why did this record not match?
- what were the top candidates?
- why was the match considered ambiguous?
- how many assets were discarded due to contributor mismatch?

Recommendation:
- add structured debug output for failed/ambiguous matches
- include top candidate titles and reasons
- include summary counts per bank:
  - extracted
  - deduplicated
  - filtered by contributor
  - strong matches
  - weak matches
  - ambiguous

8. The branch feels like a feature spike that needs narrowing before hardening.

This branch introduced many capabilities at once:
- crawler
- session saver
- config defaults
- bank adapters
- public discovery support
- dry-run logging

Recommendation:
- stabilize 2-3 banks fully first
- prove end-to-end accuracy
- only then broaden bank coverage

2.4. Most likely sources of "still not functional" behavior

If the branch is currently perceived as not fully functional, the most likely causes are:

- anti-bot / challenge pages causing incomplete crawls
- overreliance on title heuristics
- unstable extraction quality across banks
- operational complexity from retry/session flow
- weak observability when matching fails

2.5. Concrete simplifications I would make first

1. Refactor runner.py into smaller functions.
2. Make automatic approval conservative:
   - strong matches only
   - weak matches only log
3. Remove implicit interactive recovery from the main automatic pass.
4. Promote API-first or official-search-first bank strategies where possible.
5. Focus on stabilizing a small set of banks before extending coverage.

==================================================
3. Research: Can official APIs or other cleaner official channels replace scraping?
==================================================

Important framing:
- The question is not "is there a contributor API?"
- The relevant question is:
  "Can I determine from the customer side whether a given asset is publicly visible?"

That means customer-side catalog/search APIs are fully acceptable, and usually better than scraping.

High-level conclusion:
- For several banks, official customer-side APIs appear to be the best long-term source.
- For some others, official public web search is a better fallback than portfolio scraping.
- For a smaller set, scraping remains the realistic fallback.

6. Important Practical Constraint: Customer accounts are not the key issue
==================================================

Question:
- Will this work without customer accounts on the photobanks?

Answer:
- Sometimes yes
- Sometimes no
- The real issue is not whether you have a customer account
- The real issue is whether the bank exposes customer-side search/catalog access via:
  - self-serve developer API
  - partner-only API
  - official public search pages

Practical implication:
- some banks may be usable through public developer registration
- some will require partnership approval
- for the rest, official public search remains the best fallback

==================================================
7. Final Recommendation
==================================================

Do not treat "public portfolio scraping" as the universal strategy.

Recommended architecture:

1. API-first where official customer-side catalog/search APIs exist.
2. Official public site search/item-page inspection where no practical API is available.
3. Portfolio scraping only as a fallback, not the default.
4. Keep approval updates conservative:
   - strong match -> update
   - weak match -> log only
   - ambiguous -> no automatic update

If I were continuing this branch, I would do the following in order:

1. Refactor runner.py into smaller testable units.
2. Introduce match confidence tiers.
3. Stop auto-approving on weak title heuristics.
4. Replace portfolio scraping with API-first logic for the highest-value banks:
   - Shutterstock
   - Adobe Stock
   - Getty/iStock
   - Freepik
   - 123RF
5. Use official search-page fallback for the rest.

End of report.
